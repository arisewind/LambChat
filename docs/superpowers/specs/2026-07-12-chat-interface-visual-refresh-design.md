# LambChat 聊天主界面视觉升级设计

## 1. 背景与目标

LambChat 已经具备成熟的聊天、Agent、工具调用、文件预览和多端能力，也已经建立语义化颜色、阴影、圆角与动效 token。本次升级不重做产品结构，不复制外部 UI Kit，而是在现有架构上统一视觉语言，使聊天主界面呈现 Claude 与 Linear 一类克制、温暖、内容优先的专业感。

升级目标：

- 提高聊天内容的可读性，让消息而非装饰成为视觉中心。
- 统一侧栏、消息流、输入框和工具卡片的层级关系。
- 降低边框、阴影和高饱和色的使用频率，形成稳定的视觉节奏。
- 保持现有业务流程、数据结构、深浅色模式和多端行为不变。
- 建立可复用规则，避免后续页面继续产生局部风格分叉。

本次不包含欢迎页、登录页、设置页和市场页面的整体改版，也不引入新的组件框架。

## 2. 现状判断

当前前端已经具备以下可复用基础：

- `frontend/src/styles/tokens.css` 中的语义化背景、文本、边框、阴影和圆角 token。
- `frontend/src/styles/chat.css` 中的输入框、工具按钮、菜单和移动端行为。
- `SessionSidebar`、`ChatView`、`ChatMessage` 与 `ChatInput` 等边界明确的聊天组件。
- 浅色、深色、移动端、桌面端和减少动态效果等既有适配能力。

主要问题不是组件不足，而是各区域同时强调自身：侧栏选中态、消息容器、工具结果和输入框使用了较多独立的边框、卡片与阴影表达，导致页面层级略显分散。升级重点应是减少视觉噪声并统一表达，而不是继续堆叠模板组件。

## 3. 方案选择

采用渐进式系统升级：保留现有布局、组件边界和交互，只调整设计 token、布局节奏及组件外观。

未采用以下方案：

- 整套套用 UI Kit：设计稿通常不能直接映射现有 React 结构，且容易与复杂工具卡片产生风格割裂。
- 重做聊天工作台：会扩大到导航架构、面板逻辑与多端交互，超出本轮视觉升级范围。

## 4. 设计原则

### 4.1 内容优先

AI 回复采用接近文档流的开放式排版，不增加大面积消息气泡。用户消息使用紧凑、低饱和的暖色气泡，以明确角色但不抢占注意力。

### 4.2 少而明确的层级

页面只保留三类主要层级：基础背景、内容表面和浮动表面。普通内容依靠间距和底色分组；只有输入框、弹层与临时浮动面板使用明显阴影。

### 4.3 温暖而克制

浅色模式使用中性暖灰，不使用纯白铺满所有区域。强调色保持低饱和，只承担选中、焦点和主要操作。深色模式使用暖黑与石墨灰，避免纯黑、霓虹和大面积辉光。

### 4.4 密度服务于任务

聊天正文保持舒展；会话列表和工具记录保持紧凑。视觉密度按任务区分，不以全局放大留白制造“高级感”。

## 5. 页面结构

### 5.1 应用壳层

- 主背景使用柔和暖灰，内容区域与侧栏通过轻微色差分区。
- 桌面端保持当前侧栏与聊天主体结构，不新增永久第三栏。
- 聊天正文设置稳定的最大阅读宽度；工具结果等宽内容可在既有规则允许时扩展。
- 移动端保持单栏与抽屉行为，避免为了桌面视觉增加移动端层级。

### 5.2 会话侧栏

- 侧栏底色比聊天主体略深一级，不依赖完整描边分隔。
- 会话组标题使用较小字号、较弱颜色和稳定的上下间距。
- 普通会话项默认无卡片边框；悬停态仅改变底色。
- 当前会话使用柔和填充、清晰文本和细小强调标识，不同时叠加粗边框与重阴影。
- 新建会话保留最高操作优先级，其余导航动作弱化。
- 未读、收藏和菜单状态继续保留，但减少常驻图标数量，次要操作在悬停或聚焦时出现。

### 5.3 消息流

- AI 消息不使用完整气泡，头像、身份和正文形成一条清晰的阅读轴。
- 用户消息采用内容自适应宽度的暖灰气泡，并设置合理最大宽度。
- 连续消息间距小于角色切换间距，借助节奏表达分组。
- Markdown 标题、列表、引用、代码块与正文共享统一的垂直节奏。
- 消息级反馈、复制和分享操作默认弱化，在悬停、键盘聚焦或触屏显式入口中可发现。
- 加载、流式输出和错误状态不改变消息主轴，避免布局跳动。

### 5.4 输入框

- 输入框是聊天页面唯一持续存在的浮动视觉锚点。
- 使用双层结构：上层为输入内容与附件，下层为工具、技能、模型、运行模式和发送操作。
- 默认使用轻边框和低阴影；`focus-within` 提升边框对比度与阴影一级。
- 工具按钮统一尺寸、圆角、图标线宽和状态表达。
- 已启用功能使用背景与文本色表达，不额外添加多个描边环。
- 空状态与有消息状态复用同一输入组件视觉规则。
- 移动端保持 16px 文本输入字号、触控目标和安全区适配。

### 5.5 工具与 Agent 工作记录

- 工具调用、文件操作、Agent 状态和审批结果统一为“工作记录卡片”。
- 卡片头部固定包含：类型图标、标题、状态、耗时或辅助信息、展开操作。
- 默认态紧凑展示摘要；展开态展示参数、结果、文件预览或错误详情。
- 成功状态主要依靠图标和文字，不铺满绿色背景；警告与错误状态保留足够对比度。
- 嵌套工具记录通过缩进和细分隔表达层级，避免多层重阴影卡片。
- 代码、日志和参数继续使用等宽字体，普通界面文字不切换为等宽字体。

## 6. 视觉规范

### 6.1 色彩

- 浅色基础背景：暖灰；内容表面：接近白色但避免刺眼纯白。
- 主文本使用暖黑，次级文本满足可读性，不以过低透明度制造层级。
- 主题强调色只用于焦点、选中和主要操作。
- 状态色同时配合图标或文字，不以颜色作为唯一信息载体。
- 深浅色均要求正文和关键控件达到 WCAG AA 对比度。

具体色值在实现阶段基于现有 `--theme-*` 与 `--color-*` token 校准，不在组件内新增散落的硬编码颜色。

### 6.2 字体与排版

- 保持现有 Inter 与系统字体栈，避免新增网络字体依赖。
- 正文以 14–16px 为主，聊天长文本保持舒适行高。
- 标题依靠字号、字重和间距建立层级，不使用全大写装饰。
- 代码、日志和技术参数使用现有等宽字体栈。

### 6.3 圆角、边框与阴影

- 小型控件主要使用 8px 左右圆角。
- 会话项、消息气泡和工具卡片主要使用 10–12px 圆角。
- 输入框与大型浮动面板可使用 14–16px 圆角。
- 普通卡片使用单层细边框或轻微底色，不同时叠加明显边框与阴影。
- 阴影分为低、中、高三级；聊天常驻界面只使用低级，弹层使用中级或高级。

### 6.4 动效

- 悬停和焦点过渡控制在 120–200ms。
- 展开、折叠和弹层可使用轻微位移与透明度变化，不使用弹跳式大幅缩放。
- 流式消息避免逐元素入场动画。
- `prefers-reduced-motion` 下关闭非必要位移与缩放。

## 7. UI Kit 使用原则

UI Store 可作为布局与视觉节奏的参考库，但资源主要面向 Figma、Sketch、Adobe XD 等设计工具，不能视为可直接安装的 React 组件。

引入外部资源时遵循以下规则：

- 优先借鉴布局、信息层级和组件状态，不复制整套品牌表达。
- 每项实际引入的图标、插画、字体或源文件都必须记录来源、作者、许可和修改范围。
- 无清晰商业使用许可的资源只作为参考，不进入仓库和发行包。
- 不引入与现有 Lucide 图标体系重复的图标包。
- 不为了贴合模板改变既有业务流程。

## 8. 实施范围与阶段

### 第一阶段：基础语言与聊天壳层

- 校准语义化背景、边框、文本、阴影和圆角 token。
- 统一应用背景、侧栏、聊天阅读宽度和区域间距。
- 调整会话项及其选中、悬停、聚焦状态。

### 第二阶段：消息与输入体验

- 统一 AI 消息、用户消息、消息操作和 Markdown 垂直节奏。
- 重整输入框上下层视觉关系及工具按钮状态。
- 验证空状态、有消息状态、附件状态与移动键盘状态。

### 第三阶段：工作记录卡片

- 为常见工具、文件和 Agent 状态建立统一卡片骨架。
- 逐步迁移既有工具项，避免一次性重写所有特殊渲染器。
- 收敛成功、执行中、失败和需审批状态的表达。

每个阶段均可独立合并和回退，不以完成后续阶段为前置条件。

## 9. 组件与数据边界

本次设计不改变 API、WebSocket 事件、会话状态和消息数据模型。视觉规则继续依附现有组件边界：

- `SessionSidebar` 负责侧栏结构和会话状态展示。
- `ChatView` 负责聊天布局、消息滚动区和输入区位置。
- `ChatMessage` 负责角色消息与消息级操作。
- `ChatInput` 及其子组件负责输入、附件、选择器和工具栏。
- 工具 item 组件负责各自内容，公共卡片骨架只提供一致外观与状态接口。

错误、加载、取消、审批和流式状态继续使用现有数据来源。视觉改造不能改变状态判定或副作用时序。

## 10. 可访问性与响应式

- 所有交互控件保留可见的 `focus-visible` 状态。
- 桌面端图标按钮最小目标为 36px；移动端主要触控目标不小于 44px。
- 颜色不是状态的唯一提示，关键状态同时提供文字或图标。
- 在 375px、768px、1024px 和 1440px 宽度验证布局。
- 文本放大、长会话标题、长模型名和多附件不能破坏布局。
- 深浅色模式均检查对比度、焦点状态和悬停之外的可发现性。

## 11. TDD 与验收标准

实现必须遵循项目的 Red-Green-Refactor 流程。优先使用纯函数和源码结构测试；需要验证交互与 DOM 时使用 Testing Library。

最低验收范围：

- token 测试验证组件不新增散落的核心颜色、阴影和圆角硬编码。
- 侧栏测试覆盖普通、悬停可用性、选中、未读和键盘聚焦状态。
- 消息测试覆盖 AI、用户、流式、错误和长内容布局。
- 输入框测试覆盖空状态、聚焦、附件、启用工具、发送中和移动端视口。
- 工具卡片测试覆盖折叠、展开、执行中、成功、失败和审批状态。
- 源码结构测试保护 `prefers-reduced-motion`、移动端安全区和 16px 输入字号。
- 运行前端测试、lint 和生产构建；视觉验收覆盖深浅色及四个目标宽度。

升级完成的判断标准：聊天内容拥有最强视觉优先级；侧栏和工具记录易于扫描但不过度抢眼；输入框清晰可发现；常见状态在键盘、触屏、深浅色和减少动效环境下均可使用；现有聊天功能与数据流没有行为回归。

## 12. 风险与控制

- **全局 token 影响非聊天页面**：先增加或校准语义 token，再逐组件迁移；避免直接改变被全站复用但语义不明确的基础变量。
- **工具卡片种类过多**：先建立骨架和高频样例，再分批迁移特殊 item。
- **仅靠截图判断导致状态遗漏**：以组件状态矩阵和自动化测试补充视觉验收。
- **外部资源授权不清**：默认只参考，不下载、不提交、不随发行包分发。
- **移动端空间不足**：不增加永久导航层级，输入框工具继续允许按优先级收纳。
